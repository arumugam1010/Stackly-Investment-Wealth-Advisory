1. Executive Summary & Market Position

Stackly Capital represents a paradigm shift in the digital wealth management sector, combining the strict accountability of a fiduciary Registered Investment Advisor (RIA) with automated, institutional-grade quantitative trading engines. The platform satisfies a critical gap in the financial services market: offering customizable portfolios to retail, high-net-worth (HNW), and family office investors without the high management fees characteristic of traditional firms.

1.1 The Fiduciary Mandate

Unlike standard robo-advisors that prioritize automated, static index fund models, Stackly's architecture is built around a true fiduciary commitment. Fiduciary duty requires that all advisors and system algorithms legally act in the absolute best interest of the client. Under Section 206 of the Investment Advisers Act of 1940, this mandate translates into two main duties: the Duty of Care and the Duty of Loyalty. Stackly enforces this programmatically through its portfolio generation and allocation protocols.

1.2 System Value Proposition

The platform eliminates the typical 1.00% Assets Under Management (AUM) advisory fee, replacing it with a predictable flat subscription model. This architecture specification outlines the technological infrastructure that supports this low-cost model, ensuring high availability, sub-second execution speeds, military-grade security, and robust integrations with custodian networks.

Fiduciary Standard: Any automated algorithm running within the Stackly core must prove that asset allocations are optimized solely based on the client's risk capacity and financial goals, without commission-incentivized fund selection.

2. System Architecture Design & Topologies

The Stackly platform employs a distributed, microservices-oriented architecture designed to handle concurrent portfolio computations, secure customer communications, and real-time market data ingestion. The system is split into three main logical tiers: Client Presentation Tier, Application Core Services Tier, and the Custodial Integration Gateway.

2.1 High-Level Architecture Topology

All client and advisor dashboards communicate with the Application Core Services Tier via an API Gateway. This gateway handles session management, rate limiting, and request routing. The microservices are orchestrated using Kubernetes, allowing individual scaling of components during high market volatility trading sessions.

Microservice Primary Responsibility Data Store Interaction
User Authentication Service OAuth2 token verification, MFA, RBAC permission resolution. Redis Session Cache, Postgres User DB
Advisory Matching Engine Matches client profiles with CFPs using specialization vectors. Postgres Advisor DB
Portfolio Engine Calculates risk matrices, updates allocations, triggers rebalancing. TimescaleDB Portfolio Ledger
Clearing House Service ACATS electronic transfer coordination, custodial sync. Queue (RabbitMQ)

2.2 Database Layer Segmentation

To ensure high performance, relational data (such as user accounts, compliance records, and messaging logs) is stored in Postgres instances utilizing read replicas. Time-series data, specifically portfolio performance metrics and historical stock valuations, is directed to TimescaleDB databases optimized for high-write loads and fast analytical aggregates.

4. Algorithmic Portfolio Optimization & Theory

Stackly's core algorithm uses a proprietary modification of Modern Portfolio Theory (MPT), originally developed by Harry Markowitz. The algorithm constructs an "efficient frontier" of portfolios that offer the maximum possible expected return for a given level of risk.

4.1 Expected Returns & Covariance Optimization

The system calculates expected returns using historical asset performance, capital market assumptions, and macroeconomic indicators. A key metric is the covariance between asset classes (e.g. Equities, Fixed Income, Alternatives). The platform leverages this to diversify portfolios and reduce systemic risk.

4.2 Rebalancing Thresholds

Portfolios will naturally drift from their target allocation over time due to market movement. The rebalancing engine checks for drift daily using two parameters: time-based triggers and tolerance bands. If an asset class drifts by more than 5.0% absolute (or 20.0% relative to its target allocation), a rebalancing transaction is queued.

Drift Calculation Formula: Drift is measured as |Current Weight - Target Weight|. If this value exceeds the threshold, the portfolio rebalancing process sells overallocated classes and buys underallocated classes to restore the target risk profile.

5. Tax-Loss Harvesting & Cost-Basis Tracking Specifications

Tax-loss harvesting (TLH) is one of the most effective strategies for accelerating wealth accumulation in taxable accounts. Stackly's Tax Optimization Engine automatically monitors portfolios for opportunities to harvest losses, which can then be offset against taxable capital gains or ordinary income.

5.1 Cost-Basis Accounting Methodologies

To implement TLH, the system must track purchase lots with precision. Stackly supports two methods: First-In-First-Out (FIFO) and Highest-In-First-Out (HIFO). For tax harvesting, the HIFO method is utilized to target the shares with the highest cost basis, maximizing the harvested capital loss.

5.2 The Wash Sale Rule

The Internal Revenue Service (IRS) Wash Sale Rule prevents investors from claiming a capital loss on the sale of a security if they buy a "substantially identical" security within 30 days before or after the sale. Stackly's system prevents this by identifying proxy assets (e.g., swapping an S&P 500 Index ETF for a Russell 1000 Index ETF) that maintain the portfolio's target market exposure while avoiding wash-sale violations.

6. Custodial Integration Protocols (ACATS, Plaid, Apex)

Stackly functions as an overlay software platform, meaning customer assets do not sit on Stackly's balance sheet. Instead, assets are held at institutional clearing custodians (such as Apex Clearing or Pershing). This division of duties minimizes operating risk and protects client capital.

6.1 Plaid & Bank Account Verification

Banking connections use Plaid link tokens. Once authenticated, the Plaid API returns a secure access token used to pull routing/account details and trigger ACH fund transfers. No bank login credentials are saved on Stackly servers.

6.2 ACATS Automated Customer Account Transfer Service

Transferring an existing account from another brokerage involves the National Securities Clearing Corporation's (NSCC) ACATS protocol. The integration coordinates the transfer of all shares, cash balances, and cost-basis logs. The process takes 3 to 6 business days and is monitored via webhooks.

Transfer Phase Webhook Event Required Verification Data
Initiation acats.initiated SSN match, account number validation, brokerage ID.
Validation acats.validated Asset list confirmation, clearing house verification.
Settlement acats.settled Electronic transfer of shares, cost-basis ledger sync.

7. Security, Access Controls (RBAC), and Fiduciary Compliance

Security is the foundation of digital wealth advisory. The platform's security framework is designed around the Principle of Least Privilege (PoLP) and matches security controls mandated by SOC 2 Type II and SEC cybersecurity frameworks.

7.1 Role-Based Access Control (RBAC) Permissions

User accounts are categorized into roles. System administrators can never view client portfolio performance or financial details, and advisors can never modify system configurations or security credentials.

7.2 Encryption Specifications

Data in transit is encrypted using TLS 1.3 with strong cipher suites. Data at rest is encrypted using AES-256-GCM. Database backups are encrypted before storage in secure, air-gapped repositories.

Fiduciary Secure Vault: Documents uploaded by clients (such as tax returns and estate trusts) are saved in an encrypted AWS S3 Bucket with Server-Side Encryption (SSE-KMS) and access logs enabled.

8. System Performance SLAs, Disaster Recovery, and Infrastructure DR

This chapter defines the system operations, performance targets, and business continuity plans that govern Stackly Capital's cloud infrastructure.

8.1 Performance SLAs

The platform targets a 99.99% system uptime SLA. API response times are monitored globally, with a target sub-100ms latency for dashboard endpoints and portfolio calculation services.

8.2 Recovery Objectives (RTO & RPO)

The platform defines strict objectives to handle database corruption or regional cloud failures:

  • Recovery Time Objective (RTO): Less than 2 hours to fully restore service in a backup region.
  • Recovery Point Objective (RPO): Less than 5 minutes of potential data loss via continuous database log replication.

In the event of a regional outage, DNS routing automatically redirects traffic to a secondary active-passive database replica and hot-standby Kubernetes cluster in an alternate geographic region, maintaining system continuity.
